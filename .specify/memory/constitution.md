# hakoview Constitution

## Core Principles

### I. コード品質と保守性
- 可読性と一貫性を最優先し、リンタと型チェックを必ず通す。
- 変更は小さく、単一責務と明確な境界を守る。
- 既存の命名規則と構成に合わせ、例外は理由を記録する。
理由: 保守コストとバグ混入を抑えるため。

### II. テスト基準は必須
- すべての機能追加とバグ修正に自動テストを含める。
- 重要なユーザー経路は統合テストで保証する。
- 回帰は再現テストを先に追加し、修正後に通す。
理由: 仕様と挙動のズレを早期に検知するため。

### III. ユーザー体験の一貫性
- コマンド名、文言、状態表示、エラーメッセージは統一する。
- 破壊的変更は移行手順と告知を用意し、UXを損ねない。
- 受け入れ条件で体験の期待値を明文化する。
理由: 学習コストと混乱を最小化するため。

### IV. パフォーマンス予算の遵守
- 機能ごとに性能目標を定義し、測定結果を添える。
- 起動や操作の体感遅延を生む処理は避け、遅延実行を使う。
- 主要経路の性能退行は許容しない。
理由: 拡張機能としての信頼性と快適さを守るため。

### V. ドキュメントは日本語で維持
- 仕様、設計、運用、ユーザー向け文書は日本語で記述する。
- 専門用語や外来語は用語集または注釈で補足する。
理由: チーム内外の理解と共有を円滑にするため。

## 品質ゲート

- 変更はコード品質、テスト、UX、性能、文書更新の各原則に適合させる。
- レビューでは「何を、なぜ、どの原則で選んだか」を説明する。
- 違反が必要な場合は代替案と影響範囲を記録する。

## 意思決定と変更管理

- 重要な技術的判断は根拠とトレードオフを記録する。
- 破壊的変更には移行計画、互換性方針、リリースノートを添える。
- 仕様と実装の差異は必ず仕様側に反映する。

## Governance

- 本憲章は他の運用規約より優先される。
- すべてのPRは憲章準拠のレビューを通過しなければならない。
- 原則に影響する技術判断は、選択理由と代替案を明示する。
- 改定は提案 → 影響評価 → 合意 → 反映の順で行う。
- バージョンはセマンティックバージョニングに従う。
- 定期的に準拠状況を確認し、逸脱があれば是正計画を作成する。
